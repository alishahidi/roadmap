## Chapter 1: Domain-Driven Design (DDD)

**Introduction:** DDD is an approach that centers software development on a deep understanding of the business domain. It provides a structured way to manage complexity and ensure the software is a true reflection of the business it serves.

### The Two Pillars of DDD: Strategic and Tactical Design

- **Strategic DDD:** Concerned with the macro-architecture. It provides tools for analyzing the business domain, breaking it down into manageable parts (Bounded Contexts), and defining the high-level relationships between them.
- **Tactical DDD:** Focuses on the micro-architecture. It offers a rich set of building-block patterns (Aggregates, Entities) for designing the objects and components within a single Bounded Context.

### Strategic DDD: Charting the Business Landscape

- **Ubiquitous Language:** A shared, rigorous language developed by the team and domain experts. It must be used consistently in all communication and in the code itself to eliminate ambiguity.
- **Bounded Context:** The core strategic pattern. A Bounded Context is an explicit boundary within which a specific domain model is consistent. Identifying these boundaries is the most reliable method for defining well-designed microservices.
- **Context Mapping:** A visualization of the relationships between Bounded Contexts, using patterns like Anticorruption Layer (ACL), Shared Kernel, and Customer/Supplier to manage integrations.

## Chapter 2: Choosing Your Architectural Style

### Microservices Architecture

An approach to developing an application as a suite of small, independently deployable services, each built around a business capability. Key characteristics include decentralized data management, organization around business capabilities, and designing for failure.

### Event-Driven Architecture (EDA)

A paradigm built around producing, detecting, and consuming events. Components are loosely coupled and communicate asynchronously, making systems highly scalable and resilient. It's a natural fit for microservices.

### Foundational Patterns for Domain Isolation

These patterns structure the code *within* a service to protect business logic from external dependencies.

- **Hexagonal Architecture (Ports & Adapters):** The core logic defines 'ports' (interfaces). External technologies are 'adapters' that implement these ports, allowing the core to be tested in isolation.
- **Clean Architecture:** Organizes the system in concentric layers with a strict "Dependency Rule" where all dependencies point inwards, toward the core business rules.
